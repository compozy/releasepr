# Release Notes Feature Design

## Problem

GitHub releases generated by pr-release only contain auto-generated changelogs from conventional commits. There is no mechanism for adding informational text, code snippets, migration guides, or custom prose to releases.

## Decision

Add a `.release-notes/` directory where contributors and maintainers create markdown files with custom release note content. Files are collected and injected into the GitHub release alongside the auto-generated changelog.

## Requirements

- Both contributors and maintainers can add release notes using the same mechanism
- Always optional — the contributor decides if a change deserves a descriptive note
- Files generated via CLI command `pr-release add-note`
- Frontmatter with `title` and `type` (informational only, does not affect version calculation)
- Content appears in a separate section after the changelog in the GitHub release
- After release, files are moved to `.release-notes/archive/vX.Y.Z/`
- Repos that don't use `.release-notes/` continue working exactly as before

## Design

### 1. CLI Command `add-note`

New subcommand:

```
pr-release add-note --title "Shared layout package" --type feature
```

Flags:
- `--title` (required): note title
- `--type` (required): `feature`, `fix`, `breaking`, `highlight`
- `--body` (optional): inline content, skips editor

Behavior:
- Generates a file in `.release-notes/` with slugified title + short timestamp (e.g., `shared-layout-package-1713100800.md`)
- Creates `.release-notes/` directory if it doesn't exist
- Opens file in `$EDITOR` if available and `--body` not provided

Generated file:

```markdown
---
title: Shared layout package
type: feature
---

<!-- Write your release note here. Supports full markdown including code blocks. -->
```

Implementation: `cmd/add_note.go` + `internal/usecase/create_release_note.go`

### 2. Collection and Aggregation

New use case `CollectReleaseNotes` in `internal/usecase/`.

Behavior:
1. Reads all `.md` files in `.release-notes/` (ignores `archive/` directory)
2. Parses YAML frontmatter from each file (extracts `title` and `type`)
3. Extracts the markdown body (everything after frontmatter)
4. Groups notes by `type` in order: `breaking` -> `feature` -> `fix` -> `highlight`
5. Returns a struct with aggregated notes and a method to render as markdown

Rendered output example:

```markdown
### Release Notes

#### Breaking Changes

##### New auth middleware
The session token format changed...

#### Features

##### Shared layout package
The new package allows sharing layouts between artifacts...

#### Highlights

##### Performance improvements
40% reduction in build time...
```

If no files exist in `.release-notes/`, returns empty string — release proceeds with only the auto-generated changelog.

Implementation: `internal/usecase/collect_release_notes.go`

### 3. Template Integration

**PR Body template** (`prepare_pr_body.go`):

The template adds `{{.ReleaseNotes}}` after `{{.Changelog}}`. The field comes pre-rendered as markdown from `CollectReleaseNotes`. If empty, nothing is added.

**RELEASE_NOTES.md**: The file that pr-release already generates includes the collected release notes after the changelog section.

**Sanitization**: Release notes pass through the existing `sanitizeChangelogContent()` which escapes HTML but preserves safe markdown.

### 4. Archival After Release

New use case `ArchiveReleaseNotes` in `internal/usecase/`.

Behavior:
1. Called within the orchestrator after generating changelog and collecting notes, when creating the commit on the release branch
2. Creates `.release-notes/archive/vX.Y.Z/` directory if needed
3. Moves (git mv) all `.md` files from `.release-notes/` to `.release-notes/archive/vX.Y.Z/`
4. Included in the same commit that updates `CHANGELOG.md` and `RELEASE_NOTES.md`

Resulting structure:

```
.release-notes/
  archive/
    v0.1.12/
      shared-layout-package-1713100800.md
      fix-execution-order-1713100900.md
    v0.1.11/
      ...
  new-note-for-next-release.md
```

Rollback: if the release fails and the saga executor rolls back, files are restored from `archive/` back to `.release-notes/` as a compensating action.

A `.gitkeep` in `.release-notes/` ensures the directory exists even when empty after archival.

### 5. Edge Cases and Validation

**`add-note` validation:**
- `--type` accepts only: `feature`, `fix`, `breaking`, `highlight`. Clear error if invalid.
- `--title` cannot be empty
- Timestamp suffix guarantees filename uniqueness

**Collection validation:**
- File without valid frontmatter: warning logged, file skipped (does not block release)
- File with empty body: warning logged, file skipped
- `.release-notes/` directory doesn't exist: proceeds normally without release notes

**Encoding:** Files read as UTF-8. Frontmatter parsed with a YAML parser.

**Backward compatibility:** Repos that don't use `.release-notes/` continue working exactly as before — collection returns empty and nothing changes.
